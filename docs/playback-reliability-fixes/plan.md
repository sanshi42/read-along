# 朗读可靠性修复 Plan

## 方案

1. 在 `MacOSSayTTS` 适配层把用于包裹正文、但不应影响朗读内容的标点统一归一为空格。归一化只作用于传给 `say` 的输入，不修改结构化正文、数据库文本或 API 响应。
2. 为 TTS 输入归一化补充覆盖中文双引号和书名号的回归测试，锁定“包裹标点不应吞掉正文”的行为。
3. 在阅读页引入浏览器端下一句音频准备：后台预取仍负责生成/缓存音频，同时为连续播放预创建下一句 `HTMLAudioElement`、设置倍速并触发加载，让 `onended` 能复用已准备的音频。
4. 为音频衔接逻辑补充可跑的前端单元测试，覆盖下一句音频缓存、切句失效和倍速同步。

## Topic 级验证

- 运行 `make check`。
- 前端交互变更完成后，在浏览器中验证连续播放主流程。

## 验证结果

- `make check` 通过。
- 已用临时 `READ_ALONG_HOME` 创建材料并预生成音频，页面能加载材料，API 与浏览器端预取能请求到全部句子音频。
- in-app Browser 在可见和后台两种状态下都拒绝 `HTMLAudioElement.play()`，未能在该环境完成真实音频连续播放验收；需要在普通浏览器中做一次人工听感确认。
