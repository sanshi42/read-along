# 朗读加载流畅性 Plan

## Approach

- 采用前端驱动的滚动窗口预取：阅读页打开后准备当前句和后续 4 句，播放、选句和连续推进时重新补齐窗口。
- 复用现有单句音频 GET 接口，不新增后端 prepare/status API，不修改数据库 schema。
- 后台预取串行执行；前台播放当前句时复用同句 in-flight 请求，避免重复生成。
- 预取只影响音频准备，不推进阅读进度，不改变当前句、朗读完成或材料正文语义。

## Decisions

- 预取锚点优先使用未完成阅读进度；无进度或已朗读完成时从第一句开始。
- 默认窗口固定为当前句加后 4 句。
- 后台未来句失败不立刻打扰用户；后台自动重试一次，仍失败后等待该句成为当前句或用户播放时在播放器显示重试。
- UI 状态集中在底部播放器，不给正文中的每句添加可见音频状态标记。

## API

- 公开后端 API 不变，继续使用 `/api/materials/{material_id}/sentences/{sentence_id}/audio`。
- 前端内部维护句子音频准备状态、错误信息、后台重试次数和 in-flight promise。

## Verification

- 前端单元测试覆盖窗口计算、已完成材料从第一句预取、窗口推进、后台失败只重试一次、前台重试重新请求。
- 运行 `make check`。
- 在浏览器中验证打开阅读页后预取、连续朗读跨句、失败提示和重试路径。

## Verification Result

- `make check` 通过，覆盖 Ruff、格式检查、Pyrefly、200 项后端测试、15 项前端测试和前端生产构建。
- 浏览器使用临时材料库打开三句测试材料，页面加载后自动串行请求 3 个句子音频，验证打开即预取和窗口补齐行为。
- 浏览器点击播放时进度 PUT 成功，且未触发新的音频准备请求；in-app Browser 环境未实际播放媒体，播放器进入现有可恢复错误状态。
- 直接请求音频 API 返回 RIFF/WAVE 头；`afinfo` 确认生成音频为 1 声道、22050 Hz、Int16 WAV。
