# 朗读可靠性修复 Plan

## 方案

1. 在 `MacOSSayTTS` 适配层删除用于包裹正文、但不应影响朗读内容的成对标点。归一化只作用于传给 `say` 的输入，不修改结构化正文、数据库文本或 API 响应。
2. 为 TTS 输入归一化补充覆盖中文双引号、书名号和连续引号内容的回归测试，锁定“包裹标点不应吞掉正文”的行为。
3. 音频缓存不再只凭 `{material_id}/{sentence_id}.wav` 命中；生成音频时写入规范化 TTS 输入指纹，复用缓存前必须比较指纹，不匹配则丢弃旧音频并重新生成。
4. 音频响应使用 `private, no-cache`，避免浏览器长期无条件复用旧音频。前端在用户点击“修复朗读”后额外使用本地 reload token 请求音频，绕过此前已经写入浏览器的旧缓存项。
5. 阅读设置面板提供“修复朗读”入口：清理当前材料的服务端音频缓存、重置句子音频状态、清空浏览器端预取队列和音频元素缓存；如果用户点击前正在播放，则修复后从当前句当前秒数继续。
6. 在阅读页引入浏览器端下一句音频准备：后台预取仍负责生成/缓存音频，同时为连续播放预创建下一句 `HTMLAudioElement`、设置倍速并触发加载，让 `onended` 能复用已准备的音频。
7. 为音频衔接、缓存清理和 reload token 逻辑补充可跑的前后端测试。

## Topic 级验证

- 运行 `make check`。
- 前端交互变更完成后，在浏览器中验证连续播放主流程。

## 验证结果

- `make check` 通过。
- 已用临时 `READ_ALONG_HOME` 创建材料并预生成音频，页面能加载材料，API 与浏览器端预取能请求到全部句子音频。
- in-app Browser 在可见和后台两种状态下都拒绝 `HTMLAudioElement.play()`，未能在该环境完成真实音频连续播放验收；需要在普通浏览器中做一次人工听感确认。
- 追加“修复朗读”后，`make check` 通过；in-app Browser 只读验证阅读设置面板显示“修复朗读”、按钮可用且无控制台错误，未点击按钮以避免清理用户真实材料缓存。
